# 分支集成管理使用指南

本文档用于说明 [git_branch_manager.py](/Users/xue/file/claude/branchManager/git_branch_manager.py) 的使用方式、适用场景和注意事项，帮助你基于统一流程完成开发分支、集成分支和发布回写操作。

## 1. 工具简介

`git_branch_manager.py` 是一个交互式 Git 分支管理脚本，主要解决以下几类问题：

- 从 `master` / `main` 快速创建规范化的开发分支
- 创建测试或发布用的集成分支
- 将多个开发分支批量合并到集成分支
- 同步已集成开发分支后续新增的提交
- 在冲突发生时引导人工解决，并支持 `rerere` 自动复用历史解决方案
- 删除本地或远端分支
- 将 `release_*` 分支合并回 `master` / `main`

## 2. 适用分支模型

脚本默认识别以下分支类型：

- 基线分支：`master` 或 `main`
- 开发分支：`feature_*`、`bugfix_*`
- 集成分支：`dev_*`、`release_*`

脚本会根据分支名前缀自动分类，因此请尽量遵循以下命名规则。

## 3. 分支命名规则

### 3.1 开发分支

脚本会自动生成如下格式的分支名：

```text
feature_<名称>_yyyyMMdd
bugfix_<名称>_yyyyMMdd
```

例如：

```text
feature_login_20260310
bugfix_order_timeout_20260310
```

说明：

- `<名称>` 由你输入
- 日期后缀由脚本自动生成
- 不允许包含 Git 分支非法字符，例如空格、`~`、`^`、`:`、`?`、`*`、`[`、`\`

### 3.2 集成分支

脚本会自动生成如下格式的集成分支名：

```text
dev_<版本或名称>_yyyyMMdd
release_<版本或名称>_yyyyMMdd
```

例如：

```text
dev_3.5.0_20260310
release_3.5.0_20260310
```

## 4. 运行方式

在目标 Git 仓库目录中执行：

```bash
python3 /Users/xue/file/claude/branchManager/git_branch_manager.py
```

启动后脚本会先做两件事：

1. 检查当前目录是否为 Git 仓库
2. 检查当前仓库是否开启 `rerere`

如果 `rerere` 未开启，脚本会提示是否现在启用。

## 5. 主菜单说明

启动后主菜单包含以下功能：

1. 创建开发分支（`feature` / `bugfix`）
2. 集成分支管理（测试 / 生产）
3. 合并集成分支到 `master`
4. 删除分支
0. 退出

界面顶部会显示当前状态：

- 当前所在分支
- 当前仓库中的开发分支数量
- 当前仓库中的集成分支数量

## 6. 功能详解

### 6.1 创建开发分支

菜单路径：

```text
主菜单 -> 1. 创建开发分支（feature / bugfix）
```

操作流程：

1. 选择分支类型：`feature` 或 `bugfix`
2. 输入开发分支名称
3. 脚本自动切换到 `master` / `main`
4. 尝试执行 `git pull origin <base>`
5. 基于最新基线创建开发分支并自动切换过去

适用场景：

- 新功能开发
- 缺陷修复

注意事项：

- 如果远端拉取失败，脚本会提示并继续使用本地基线
- 如果切换到 `master` / `main` 失败，脚本会中止，不会在错误分支上继续创建

### 6.2 创建集成分支

菜单路径：

```text
主菜单 -> 2. 集成分支管理 -> 1. 创建集成分支
```

操作流程：

1. 选择集成用途：`dev` 或 `release`
2. 输入版本号或名称
3. 脚本切换到 `master` / `main`
4. 尝试同步基线
5. 创建新的集成分支
6. 选择要合并进去的开发分支
7. 批量执行合并
8. 写入一条追踪提交，用于记录本次实际成功集成的开发分支

追踪提交格式如下：

```text
[DREO-MERGE] <集成分支> <- branch1,branch2,...
```

这个追踪提交用于后续“同步更新集成分支”功能识别哪些开发分支已经集成过。

### 6.3 更新集成分支

菜单路径：

```text
主菜单 -> 2. 集成分支管理 -> 2. 更新集成分支（同步已集成分支新提交）
```

作用：

- 对已经集成过的开发分支重新执行合并
- 只引入这些开发分支在上次集成之后新增的提交

脚本处理逻辑：

1. 根据 `[DREO-MERGE]` 追踪提交找出该集成分支历史上集成过的开发分支
2. 检查这些分支是否仍存在于本地
3. 对每个仍存在的分支计算新增提交数
4. 无新增提交则跳过
5. 有新增提交则执行 re-merge
6. 对本次实际成功同步的分支重新写一条追踪提交

适用场景：

- 某个功能分支在第一次集成后又补了几个提交
- 测试环境需要持续跟进多个开发分支的最新进展

### 6.4 添加新的开发分支到集成分支

菜单路径：

```text
主菜单 -> 2. 集成分支管理 -> 3. 添加新的开发分支到集成分支
```

作用：

- 给已有的 `dev_*` 或 `release_*` 集成分支补充新的开发分支

脚本会自动排除已经通过追踪提交识别为“已集成”的开发分支，只展示还未集成的候选分支。

### 6.5 合并发布分支回 master

菜单路径：

```text
主菜单 -> 3. 合并集成分支到 master
```

说明：

- 这里只允许选择 `release_*` 分支
- 不会把 `dev_*` 测试分支直接合回主干

操作流程：

1. 选择要回写的 `release_*` 分支
2. 脚本切换到 `master` / `main`
3. 尝试同步基线
4. 执行合并
5. 输出最近 5 条提交记录
6. 提示你执行 `git push origin <base>`

适用场景：

- 预发布验证完成后，将发布分支正式回写到主干

### 6.6 删除分支

菜单路径：

```text
主菜单 -> 4. 删除分支
```

子菜单：

- 仅本地
- 本地 + 云端

脚本特点：

- 自动保护当前分支和 `master` / `main`
- 按“开发分支 / 集成分支 / 其他分支”分类展示
- 支持多选和 `all`
- 默认先尝试安全删除：`git branch -d`
- 如果存在未合并提交，会二次确认是否强制删除：`git branch -D`

如果你选择“本地 + 云端”，脚本还会尝试执行：

```bash
git push origin --delete <branch>
```

## 7. 冲突处理说明

当合并过程中出现冲突时，脚本会进入冲突处理菜单。

### 7.1 手动解决冲突

脚本会提示你：

1. 编辑冲突文件
2. 删除冲突标记
3. 执行 `git add <文件>`
4. 回到脚本中选择“继续合并”

脚本会检查：

- 是否还有未解决冲突文件
- 是否可以成功提交 merge commit

### 7.2 放弃本次合并

如果你不想继续本次合并，可以在冲突处理菜单中选择放弃，脚本会执行：

```bash
git merge --abort
```

### 7.3 rerere 自动复用

如果仓库启用了 `rerere`，在同样的冲突再次出现时：

- Git 可能自动复用之前保存的解决结果
- 脚本会检查未合并文件中是否仍然存在冲突标记
- 如果内容已经被自动解决，脚本会自动 `git add` 并自动完成提交

这适合重复集成、重复回写、同一冲突反复出现的场景。

## 8. 典型使用流程

### 场景一：新功能开发并进入测试环境

推荐流程：

1. 从主菜单创建开发分支
2. 在对应开发分支完成开发和提交
3. 进入“集成分支管理”
4. 创建 `dev_*` 分支
5. 选择多个开发分支一起合入
6. 在测试环境验证

### 场景二：开发分支补充提交后同步到测试环境

推荐流程：

1. 开发人员继续在原 `feature_*` 分支上提交
2. 测试人员进入“更新集成分支”
3. 选择目标 `dev_*` 分支
4. 脚本自动识别哪些分支有新增提交
5. 只对有变化的分支重新合并

### 场景三：测试完成后发版并回写主干

推荐流程：

1. 创建 `release_*` 分支并完成发布验证
2. 在主菜单选择“合并集成分支到 master”
3. 选择目标 `release_*`
4. 合并完成后手动推送到远端

## 9. 常用输入规则

脚本中常见输入方式如下：

- 单选：输入数字编号
- 多选：输入 `1,2,3`
- 全选：输入 `all`
- 返回上一级：输入 `0`
- 确认操作：输入 `y`
- 取消操作：输入 `n`

## 10. 注意事项

### 10.1 需要在 Git 仓库中运行

如果当前目录不是 Git 仓库，脚本会直接退出。

### 10.2 建议先提交或清理本地改动

虽然脚本会保护关键流程，但如果你在脏工作区中执行复杂合并，仍然容易混入无关变更。建议在使用前先：

- 提交当前修改
- 或 stash 暂存当前修改

### 10.3 追踪提交不要随意删除

`更新集成分支` 和 `添加新的开发分支到集成分支` 依赖 `[DREO-MERGE]` 空提交识别历史。如果你删除了这些提交，脚本对“哪些分支已经集成”的判断会不准确。

### 10.4 集成分支同步依赖本地分支存在

如果某个曾经集成过的开发分支已经被本地删除，脚本会提示并跳过该分支。

### 10.5 回写主干后仍需手动推送

脚本在“合并发布分支回 master”完成后只会提示推送命令，不会自动执行远端推送。

## 12. 常见问题

### Q1：脚本提示拉取远端失败，还能继续吗？

可以。脚本会提示你当前改为使用本地 `master` / `main` 继续。是否继续取决于你是否接受本地基线不是最新状态。

### Q2：为什么更新集成分支时有的开发分支被跳过？

通常有两种原因：

- 该分支没有新增提交
- 该分支在本地已不存在

### Q3：为什么某个分支没有出现在“添加新的开发分支到集成分支”列表里？

因为脚本认为它已经集成过。这个判断基于 `[DREO-MERGE]` 追踪提交。

### Q4：冲突自动解决后为什么还要检查文件？

即使 `rerere` 自动复用了历史方案，也建议你快速检查一下自动解决结果是否符合当前业务预期。

## 13. 总结

如果你的团队采用“开发分支 -> 集成分支 -> 发布分支 -> 主干回写”的工作模式，这个脚本可以明显减少手工输入 Git 命令的成本，并把重复性分支操作标准化。

推荐做法是：

- 统一分支命名
- 保留追踪提交
- 尽早开启 `rerere`
- 在正式发布前先通过 `dev_*` 或 `release_*` 做完整验证
