# Dreo Branch Manager 使用指南

## 1. 为什么设计此脚本

### 1.1 它解决了什么问题

日常 Git 分支管理里，最容易出问题的不是单条命令，而是整条流程：

- 开发分支命名不统一
- 测试分支和发布分支混在一起，哪些功能已经集成过不清楚
- 集成分支长期不跟 `master` 同步，提测环境与真实主干脱节
- 某个功能临时不上线时，不知道该如何灵活调整集成分支
- 过了一段时间后，已经很难还原“这个集成分支到底按什么顺序处理过哪些分支”

`dreo_branch_manager.py` 的目标就是把这套流程变成可重复、可追踪、可回放的标准动作，而不是依赖个人记忆。

### 1.2 这个脚本的优点

#### 1. 灵活的 feature 分支控制

你不需要把所有开发分支一次性打包处理。脚本支持：

- 创建标准化的 `feature_*` / `bugfix_*` 分支
- 创建集成分支时自由选择需要集成的开发分支
- 后续随时往已有集成分支里追加新的开发分支
- 同步时只处理已经集成过且确实有新增提交的分支

这意味着你可以非常灵活地控制哪些功能要进入当前测试、哪些功能暂时不上线、哪些功能后续再补进来。

#### 2. `master` 自动合并

这是这套脚本最重要的设计点之一。

在集成分支创建和更新过程中，脚本会主动围绕 `master` / `main` 做同步：

- 创建集成分支时，总是基于最新 `master` / `main`
- 更新集成分支时，会先把最新 `master` / `main` 合入当前集成分支
- 只有主干同步完成后，才继续处理开发分支的新增提交

这样可以避免“测试分支已经很久没跟主干同步，最后发布时才发现差异过大”的问题。

#### 3. 可按需要控制上线、下线和范围

很多时候，真正需要的不是“把所有功能都合进去”，而是：

- 只带某几个功能去测试
- 某个功能先不上线
- 某个功能改完后再单独补进已有集成分支
- 发布时只允许 `release_*` 回写到主干

脚本通过“可选择的 feature 分支集成 + 追踪提交 + 集成分支分层”把这件事做成了明确流程。  
你可以把它理解成：脚本在帮助你管理“本次到底要带哪些功能上线”。

#### 4. 冲突处理更稳定

脚本内置了冲突处理引导，并支持 `rerere`：

- 冲突时会进入明确的处理菜单
- 可以选择继续解决或直接放弃本次合并
- 在重复冲突场景下，`rerere` 可以自动复用历史解决方案

#### 5. 有记录、可复盘

每次成功集成的开发分支，都会写入：

```text
[DREO-MERGE] <集成分支> <- branch1,branch2,...
```

这不仅用于后续“更新集成分支”的自动识别，也让你可以生成分支处理报告，复盘整个集成顺序和分支流转过程。

## 2. 如何安装此脚本

### 2.1 安装前提

- 操作系统：macOS / Linux
- 已安装 `python3`
- 已安装 `git`

### 2.2 执行安装器

在项目目录中执行：

```bash
python3 /Users/xue/file/claude/branchManager/dreo_branch_install.py
```

安装器会弹出交互菜单：

```text
1. 安装   （将当前目录下的 dreo_branch_manager.py 安装到系统）
2. 更新   （更新已安装的 dreo_branch_manager.py 文件）
3. 卸载   （完全卸载已安装文件与配置，不删除当前目录源码）
0. 退出
```

### 2.3 安装

选择 `1. 安装` 后，安装器会执行以下动作：

- 把当前目录中的 [dreo_branch_manager.py](/Users/xue/file/claude/branchManager/dreo_branch_manager.py) 复制到用户目录
- 生成 3 个命令入口：
  - `dreo_branch_manager`
  - `branch`
  - `dbm`
- 自动为常见 shell 写入 `PATH` 配置

默认安装位置：

- 主脚本：`~/.local/share/dreo_branch_manager/dreo_branch_manager.py`
- 命令目录：`~/.local/bin`

安装完成后，你可以在任意 Git 仓库目录中直接执行：

```bash
dreo_branch_manager
```

或：

```bash
branch
dbm
```

### 2.4 当前终端立即生效

安装器会自动写入 shell 配置，但它不能直接修改“当前这个父终端”的环境变量。  
如果你希望安装后当前终端立即可用，请执行安装器输出的激活命令，例如：

```bash
export PATH="$HOME/.local/bin:$PATH"
```

或者执行：

```bash
source ~/.zshrc
```

也可以重新打开终端窗口。

### 2.5 更新

当你修改了主脚本后，可以再次执行：

```bash
python3 /Users/xue/file/claude/branchManager/dreo_branch_install.py
```

选择 `2. 更新`，安装器会：

- 覆盖已安装的 `dreo_branch_manager.py`
- 保留原有命令入口
- 同步修正 PATH 配置块

### 2.6 卸载

选择 `3. 卸载` 后，安装器会：

- 删除已安装的命令入口
- 删除复制到用户目录的主脚本
- 清理 shell 中由安装器写入的 PATH 配置块
- 保留当前目录中的源码脚本，不会误删你的工作目录文件

## 3. 脚本使用指引

### 3.1 推荐使用方式

进入任意 Git 仓库目录后执行：

```bash
dreo_branch_manager
```

或：

```bash
branch
```

主菜单当前包含以下功能：

1. 创建开发分支（`feature / bugfix`）
2. 集成分支管理（测试 / 生产）
3. 合并集成分支到 `master`
4. 生成分支处理报告
5. 删除分支
0. 退出

### 3.2 建议的日常流程

推荐把这套工具按下面顺序使用：

1. 从 `master` / `main` 创建开发分支
2. 完成功能开发后，创建 `dev_*` 集成分支做联调或测试
3. 测试过程中如果 `master` 有新增代码，使用“更新集成分支”同步主干并继续同步开发分支新增提交
4. 某些功能需要后补时，使用“添加新的开发分支到集成分支”
5. 准备发布时，创建或维护 `release_*` 分支
6. 验证通过后，将 `release_*` 合并回 `master`
7. 最后按需生成一份分支处理报告留档

### 3.3 创建开发分支

菜单路径：

```text
主菜单 -> 1. 创建开发分支（feature / bugfix）
```

脚本行为：

- 自动识别 `master` / `main`
- 先切换到基线分支并尝试同步
- 生成标准命名：

```text
feature_<名称>_yyyyMMdd
bugfix_<名称>_yyyyMMdd
```

适用场景：

- 新功能开发
- Bug 修复

### 3.4 集成分支管理

菜单路径：

```text
主菜单 -> 2. 集成分支管理（测试 / 生产）
```

#### 1. 创建集成分支

适用于：

- 拉起一个新的测试分支 `dev_*`
- 拉起一个准备发布的 `release_*`

关键特性：

- 基于最新 `master` / `main` 创建
- 可自由选择本次要纳入的开发分支
- 成功集成后写入追踪提交

#### 2. 更新集成分支

这是最关键的一个入口。

脚本会按下面顺序处理：

1. 找到该集成分支历史上已经集成过的开发分支
2. 先同步最新 `master` / `main`
3. 再逐个同步这些开发分支的新增提交
4. 对无新增提交的分支自动跳过
5. 只为本次成功同步的开发分支写新的追踪提交

这一步正是“`master` 自动合并”的核心实现。

#### 3. 添加新的开发分支到集成分支

适用于：

- 已有测试分支需要追加新的功能
- 某个功能之前没进本轮测试，现在需要补进去

脚本会自动排除已经识别为“已集成”的开发分支，因此不会把候选列表弄得很乱。

### 3.5 合并发布分支到 master

菜单路径：

```text
主菜单 -> 3. 合并集成分支到 master
```

行为特点：

- 只允许选择 `release_*`
- 不会误把 `dev_*` 测试分支直接并回主干
- 合并前会先切换并同步 `master` / `main`

适用场景：

- 发布前最终回写主干

### 3.6 生成分支处理报告

菜单路径：

```text
主菜单 -> 4. 生成分支处理报告
```

执行后会在当前 Git 仓库目录生成：

```text
branch_merge_report.md
```

报告内容包括：

- 当前仓库分支概览
- 推断的分支处理顺序
- Mermaid 时间线图
- Mermaid 分支流转图
- `[DREO-MERGE]` 追踪提交列表

适用场景：

- 复盘测试分支是如何形成的
- 给团队同步当前分支处理顺序
- 留档上线前的分支处理轨迹

### 3.7 删除分支

菜单路径：

```text
主菜单 -> 5. 删除分支
```

支持：

- 仅删除本地
- 同时删除本地和远端

保护规则：

- 当前分支不可删
- `master` / `main` 不可删

## 4. 常见问题

### 4.1 为什么我安装完成后，当前终端里还是提示命令不存在？

因为安装器只能修改 shell 配置文件，不能直接改掉当前父终端环境。  
请执行：

```bash
export PATH="$HOME/.local/bin:$PATH"
```

或者：

```bash
source ~/.zshrc
```

### 4.2 为什么脚本建议开启 rerere？

因为它可以记住你对同一类冲突的处理方式。  
第一次手工解决后，后续相同冲突可能自动复用结果，特别适合重复同步集成分支的场景。

### 4.3 为什么更新集成分支时，会先去合并 master？

这是有意设计的。  
目的不是“多做一步”，而是让集成分支始终尽量接近当前真实主干，避免测试通过后，真正发布时再因为主干差异出现问题。

### 4.4 如何控制某个功能本次不上线？

创建一个新的集成分支，废弃之前的集成分支即可。
也就是说，这个工具不是强制把所有 feature 一锅端，而是允许你按需控制本轮上线范围。
